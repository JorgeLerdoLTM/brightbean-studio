# Agent Integration — Public REST API

BrightBean Studio exposes a small, bearer-token-authenticated REST API at
`/api/v1/` so an external agent (a content-generation Claude Code session,
an n8n flow, a custom Python worker) can push generated images, videos, and
draft posts directly into a workspace's media library and composer.

Built-in design choices:

- **Per-workspace OR per-organization API keys.** Workspace keys write into
  that workspace's library. Org-shared keys write into the org's shared
  library (`workspace=NULL` assets visible to every workspace in the org).
  Cross-tenant access is impossible — the key's organization is the source
  of truth and any URL parameter that disagrees is rejected with 403.
- **Tokens stored hashed.** The full token is shown to a human exactly
  once at creation and never persisted anywhere queryable. Rotate by
  creating a new key and revoking the old one.
- **600 requests / hour per token.** `django-ratelimit` enforces this on
  every endpoint; the limit is checked before authentication, so it
  protects against brute-force lookup attempts too.
- **No new env vars.** API keys live in the database. The Resend / S3 /
  social-platform credentials you already configured are reused.

---

## Enable in your deploy

Migrations apply automatically on container start (the `Dockerfile` runs
`python manage.py migrate --noinput` before launching Gunicorn). When you
deploy the commit that introduces this feature, the `api_keys` table is
created on the first restart. No further action needed.

If you self-host without the bundled Dockerfile, run migrations manually:

```bash
python manage.py migrate
```

## Create an API key

### From the CLI (Railway / Heroku / Render / Docker)

```bash
# Workspace-scoped (recommended for per-brand agents)
railway run python manage.py create_api_key \
  --org "Figus" \
  --workspace "Printio" \
  --name "Printio content agent"

# Org-shared (one agent writing into every workspace's shared library)
railway run python manage.py create_api_key \
  --org "Figus" \
  --name "Org-wide automation"
```

The command prints the full token exactly once. **Copy it before pressing
anything else.** If you lose it, revoke the key and mint a new one.

If org or workspace names are not unique (e.g. multiple test orgs), pass
`--org-id <uuid>` / `--workspace-id <uuid>` instead of the name.

> Heroku users substitute `heroku run`; Render users use the Shell tab;
> Docker users `docker compose exec app python manage.py create_api_key ...`.

## Hand the key to your agent

Three ways to get the token onto the agent's machine:

1. **Environment variable** (simplest):
   ```bash
   export BRIGHTBEAN_URL=https://socials.figus.ai
   export BRIGHTBEAN_API_KEY=bbs_<the_full_token>
   ```
2. **`.env` file** the agent reads via `python-dotenv` or similar:
   ```env
   BRIGHTBEAN_URL=https://socials.figus.ai
   BRIGHTBEAN_API_KEY=bbs_<the_full_token>
   ```
3. **Secret manager** (1Password CLI, `pass`, AWS Secrets Manager): store
   under a name like `brightbean-agent/printio-key` and have the agent
   read it on startup with `op read op://Personal/brightbean-agent/printio-key`.

The bundled Claude Code skill at `docs/agent-skill/` handles all three
options automatically — see "Install the agent skill" below.

## Endpoint reference

### Identity

`GET /api/v1/whoami/` — confirms the token works and reports its scope.

```bash
curl -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
     $BRIGHTBEAN_URL/api/v1/whoami/
```

Response:

```json
{
  "scope": "workspace",
  "organization_id": "uuid",
  "organization_name": "Figus",
  "workspace_id": "uuid",
  "workspace_name": "Printio",
  "scopes": ["media:read", "media:write", "posts:write"],
  "key_name": "Printio content agent",
  "key_prefix": "bbs_kx9Lp2v8"
}
```

For an org-shared key, `scope` is `"organization"` and `workspace_id` /
`workspace_name` are `null`.

### Media — workspace scope

Base path: `/api/v1/workspaces/<workspace_id>/media/`

#### Upload an asset

```bash
curl -X POST -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  -F "file=@./cat.jpg" \
  -F "alt_text=A black cat on a red rug" \
  -F "title=Cat hero shot — campaign A" \
  -F "tags=campaign-a,cats,hero" \
  -F "source=agent" \
  -F "source_url=https://example.com/generation/abc123" \
  -F "attribution=Generated by Stable Diffusion XL 1.0" \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/media/assets/
```

Returns `201` with the asset JSON, including `processing_status: "PENDING"`.
Poll the asset until `processing_status` flips to `"COMPLETED"` and a
`thumbnail_url` populates.

Limits: 20 MB images, 1 GB video, supported MIME types are the same as the
browser uploader (`image/jpeg`, `image/png`, `image/webp`, `image/gif`,
`image/svg+xml`, `video/mp4`, `video/quicktime`, `video/x-msvideo`,
`video/webm`, `application/pdf`).

#### List assets

```bash
curl -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  "$BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/media/assets/?media_type=image&tag=campaign-a&page=1&page_size=50"
```

Query params: `folder_id` (`null` for unfoldered), `media_type` (`image`,
`video`, `gif`, `document`), `tag`, `search`, `page`, `page_size` (max 200).

#### Get / update / delete a single asset

```bash
# GET
curl -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/media/assets/$ASSET_ID/

# PATCH (updates alt_text / title / tags / folder_id)
curl -X PATCH -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"alt_text":"Updated","tags":["new","tags"]}' \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/media/assets/$ASSET_ID/

# DELETE
curl -X DELETE -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/media/assets/$ASSET_ID/
```

`DELETE` returns 409 if the asset is referenced by a scheduled post (the
response body lists the post IDs).

#### Folders

```bash
# List
curl -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/media/folders/

# Create (max 3 levels deep)
curl -X POST -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"name":"Campaign A","parent_folder_id":null}' \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/media/folders/
```

### Media — organization scope (shared library)

Identical surface at `/api/v1/organizations/<org_id>/media/...`. Requires
an org-shared key (one whose `workspace` is `null`). Assets created via
these endpoints have `workspace=NULL` and are visible to every workspace
in the org.

### Composer — draft posts (workspace scope)

#### Create a draft

```bash
curl -X POST -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "caption": "New product drop tomorrow. Link in bio.",
    "title": "Spring drop teaser",
    "asset_ids": ["uuid-of-image-1", "uuid-of-image-2"],
    "platforms": ["instagram_login", "linkedin_company"],
    "platform_captions": {
      "linkedin_company": "Excited to announce our spring collection. Available tomorrow."
    },
    "tags": ["product-launch", "spring-2026"],
    "schedule_at": null
  }' \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/posts/drafts/
```

Returns `201` plus the draft JSON including a `compose_url` deep-link
into the studio UI so a human can review and schedule:

```json
{
  "id": "uuid",
  "workspace_id": "uuid",
  "caption": "...",
  "platforms": [
    {"platform": "instagram_login", "social_account_id": "uuid", "account_name": "@figus", "status": "draft"},
    {"platform": "linkedin_company", "social_account_id": "uuid", "account_name": "Figus", "status": "draft"}
  ],
  "media": [{"asset_id": "uuid", "position": 0}, {"asset_id": "uuid", "position": 1}],
  "compose_url": "/workspace/<ws_id>/compose/<post_id>/",
  "status": "draft"
}
```

If any requested platform has no connected social account in the workspace,
the response is `422 no_connected_account_for_platforms` with the unmatched
list. Connect the missing account in the studio first.

`platform_captions` is optional; supply only the platforms that need a
caption different from the base `caption`. The composer's existing
per-platform-override logic handles fallback automatically.

`schedule_at` is optional ISO-8601 (with timezone). When null, the draft
sits in the composer waiting for a human to pick a slot. When provided,
each PlatformPost's `scheduled_at` is set to that time.

#### List / get / delete drafts

```bash
# List
curl -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/posts/drafts/

# Get
curl -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/posts/drafts/$POST_ID/

# Delete (only while still in a draft-ish status)
curl -X DELETE -H "Authorization: Bearer $BRIGHTBEAN_API_KEY" \
  $BRIGHTBEAN_URL/api/v1/workspaces/$WS_ID/posts/drafts/$POST_ID/
```

Delete returns 409 if the post is already scheduled/published.

## Install the agent skill

The repo ships a Claude Code skill at [`docs/agent-skill/`](./agent-skill/)
that teaches a separate Claude Code instance how to use this API. To install:

```bash
# Copy the skill folder into the receiving Claude Code session's skills dir
cp -r docs/agent-skill ~/.claude/skills/brightbean-studio-agent
```

Then, in the receiving Claude Code session, you can say things like:

- "Upload `cat.jpg` to my Brightbean media library with alt-text 'a cat on a rug'."
- "Generate three Instagram captions for this image, then create a draft post."
- "Push these four images and a caption to BrightBean as a draft post for Instagram and LinkedIn Company."

The skill walks the agent through reading `BRIGHTBEAN_API_KEY` / `BRIGHTBEAN_URL`,
calling `/whoami` to validate, uploading, polling processing, and creating drafts.

See [`docs/agent-skill/README.md`](./agent-skill/README.md) for details.

## Security best practices

- **One key per agent.** Don't share a single key across multiple agents
  or environments. Revocation becomes irreversible loss of access for
  everything when keys are shared.
- **Rotate periodically.** No automatic expiry by default; create a new
  key, switch the agent, revoke the old one.
- **Scope minimally.** Use workspace keys unless the agent genuinely needs
  to write into every workspace's shared library.
- **Never commit a key.** Use `.env` files in `.gitignore`'d locations, or
  a secret manager.
- **HTTPS only.** Your deploy already terminates HTTPS at the edge (Caddy
  / platform load balancer). Make sure `BRIGHTBEAN_URL` always starts
  with `https://`.

## Troubleshooting

| Status | Body | Meaning | Fix |
|---|---|---|---|
| 401 | `missing_or_malformed_authorization` | No `Authorization: Bearer ...` header | Set the header on every request. |
| 401 | `invalid_token` | Token's hash doesn't match any row | Token wrong, mistyped, or revoked. Re-mint. |
| 401 | `expired_token` | Token past `expires_at` | Mint a new key. |
| 403 | `workspace_key_required` | Org-shared key calling a workspace endpoint | Use the right endpoint family for your key. |
| 403 | `org_key_required` | Workspace key calling a shared-library endpoint | Same as above. |
| 403 | `forbidden_scope` | Workspace/org ID in URL doesn't match the key | Either fix the URL, or use a different key. |
| 422 | `file_required` | Multipart body had no `file` field | Add `-F "file=@..."`. |
| 422 | `validation_failed` | MIME or size invalid | Convert / shrink. See limits above. |
| 422 | `folder_not_found` | `folder_id` doesn't exist in scope | Use the folders list endpoint to find a valid one. |
| 422 | `no_connected_account_for_platforms` | Some `platforms[]` have no connected SocialAccount | Connect the missing account in the studio first. |
| 409 | `asset_in_use` | DELETE rejected because asset is on a scheduled post | Unschedule the post first, then delete. |
| 429 | (none, blocked) | Over 600 req/h on this token | Back off for 60s; restructure to fewer calls. |

For server errors (5xx), check the Railway / Heroku / Render logs and
correlate by the `key_prefix` you see in the request — only that prefix
is logged, never the full token.

## What's next (v2 roadmap)

These are documented in the original implementation plan as deferred
work; ask if you want any of them prioritised:

- **Studio-side AI fallback.** When the agent uploads an asset without
  `alt_text`, the studio calls Claude Vision to generate one. Same idea
  for auto-tagging.
- **Auto-draft mode.** Upload an asset alone (no caption) and the studio
  auto-creates a draft post with an AI-generated caption per target
  platform.
- **Presigned-URL upload path** for files >100 MB (avoids streaming
  through Django).
- **Outbound webhooks** so the studio can notify the agent when
  processing completes, replacing client-side polling.
- **MCP server** wrapping these REST endpoints as native Claude Code
  tools — removes the need for the agent-side skill.
